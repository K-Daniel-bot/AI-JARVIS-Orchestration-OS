# 버그 수정 리뷰

## 사용 시점

- 버그 수정 ChangeSet 리뷰
- 테스트 실패 → 수정 → 재테스트 사이클
- Planner가 test-build 피드백으로 수정 계획 생성 시

## 절차

1. 버그 이해:
   - 테스트 실패 또는 버그 리포트 읽기
   - 근본 원인 파일/라인 식별
   - 분류: 타입 에러, 로직 에러, 의존성 문제, 레이스 컨디션
2. 수정 리뷰:
   - 수정이 최소한인가? (문제 해결에 필요한 가장 작은 변경)
   - 새로운 위험 도입 여부? (보안 체크리스트 실행)
   - 기존 코드 패턴 준수 여부?
   - 타입 유지 여부? (`any` 미도입)
3. 테스트 커버리지 검증:
   - 이 버그에 대한 회귀 테스트 존재?
   - 수정 없이 테스트가 실제 실패하는지?
   - 엣지 케이스 커버 여부?
4. 부작용 검사:
   - 수정된 함수의 다른 호출자 검색
   - Public API 브레이킹 체인지 없는지 확인
   - import/export 일관성 확인
5. 리뷰 판정:
   - **PASS**: 수정 정확, 최소, 테스트 완비
   - **NEEDS_FIX**: 문제 발견 (블로커 목록)
   - **REJECT**: 근본적 접근 오류 (이유 설명)
